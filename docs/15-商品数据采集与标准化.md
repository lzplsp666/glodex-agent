# 商品数据采集与标准化

## 定位

`app/ingest` 是商品数据采集包，负责从订单侠等上游 API 获取国内电商商品数据，并统一成项目内部稳定结构。

Agent 工具层不直接调用订单侠：

```text
订单侠 API -> app.ingest 标准化 -> 商品库 / 商品索引 -> item_search -> AgentLoop
```

## 文件职责

| 文件 | 职责 |
| --- | --- |
| `app/ingest/dingdanxia.py` | 订单侠 API 原始请求客户端 |
| `app/ingest/schemas.py` | 统一入库商品结构 |
| `app/ingest/normalizer.py` | 不同平台字段映射到统一结构 |
| `app/ingest/writer.py` | 第一版写 JSONL 商品索引 |

## 统一商品结构

核心结构是 `NormalizedProduct`：

- `item_id`
- `platform`
- `title`
- `price_cny`
- `coupon_cny`
- `final_price_cny`
- `shipping_fee_cny`
- `free_shipping`
- `eta_days`
- `shop_name`
- `shop_type`
- `rating`
- `sales`
- `image_url`
- `url`
- `category`
- `attributes`
- `tags`
- `raw`

`raw` 必须保留上游原始结果，方便后续补字段和排查映射问题。

## 为什么要标准化

不同平台字段不一致。例如：

- 淘宝 / 天猫可能使用 `num_iid`、`title`、`pict_url`、`zk_final_price`、`volume`。
- 京东可能使用 `skuId`、`skuName`、`imageUrl`、`price`、`shopName`。
- 拼多多可能使用 `goods_id`、`goods_name`、`goods_thumbnail_url`、`min_group_price`、`sales_tip`。

下游的 `item_search`、`item_picker` 和 `shopping_summary` 不能直接依赖这些原始字段，所以必须统一。

## 后续扩展

当前第一版可以写入 `ITEM_SEARCH_INDEX_PATH` 指向的 JSONL 文件。后续接入 Milvus、Elasticsearch 或数据库时，继续复用 `NormalizedProduct`，只替换写入层。
