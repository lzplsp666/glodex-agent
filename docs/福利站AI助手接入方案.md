# 福利站 AI 助手接入方案

## 1. 文档目的

本文用于说明员工福利站项目的业务定位，以及将当前 Glodex Agent 项目接入福利站的可行方案。

本文中的“福利站”是对企业员工福利平台的统称，具体产品名称可能是员工福利站、A+ 福利站、关爱平台或企业内部福利商城。不同企业的实际名称、入口和功能可能存在差异。

## 2. 福利站项目是什么

福利站本质上是一个面向企业员工的数字化福利服务平台。

企业通过福利站统一管理员工福利、积分、福利额度、商品、活动和订单；员工通过员工端查看福利、使用积分或福利额度兑换商品和服务。

福利站通常不是面向社会公众开放的普通电商平台，而是服务于特定企业及其员工的企业福利平台。

### 2.1 业务参与方

```text
企业客户
  ├── HR / 行政：配置福利规则、员工和预算
  ├── 工会：组织福利活动、节日福利和员工关怀
  └── 采购人员：管理商品、供应商和结算

员工
  └── 查看福利、使用积分、兑换商品、查询订单

福利平台运营方
  ├── 商品和供应商运营
  ├── 订单、物流和售后
  ├── 客服服务
  └── 平台技术和数据运营
```

### 2.2 为什么需要福利站

传统员工福利发放通常存在以下问题：

- 福利采购、发放和统计依赖人工；
- 不同地区和不同员工的需求难以统一满足；
- 工会、HR、采购和财务之间数据分散；
- 商品选择少，员工参与度和满意度有限；
- 订单、库存、物流和售后难以统一管理；
- 企业无法实时了解福利预算和使用效果。

福利站通过平台化方式，把员工福利的配置、发放、兑换、供应链和服务集中起来，降低福利管理成本，提高员工福利体验。

## 3. 福利站的基本业务模式

福利站可以理解为“企业发放积分，员工自主兑换，平台负责履约”的业务模式。

```text
企业配置福利规则
        ↓
企业向员工发放积分或福利额度
        ↓
员工进入福利商城
        ↓
员工选择商品或服务
        ↓
系统校验积分、库存和兑换资格
        ↓
创建兑换订单并扣除积分
        ↓
供应商发货或提供服务
        ↓
员工查询物流、完成售后
```

福利商品可能来自多个供应商、品牌商、电商渠道和本地服务商。平台对员工展示统一的福利商品池，供应商和来源信息主要由后台负责管理。

因此，福利站不一定自己生产或囤积所有商品，也不一定把所有商品直接展示为京东、淘宝或拼多多商品。外部渠道更多是后台供应链来源，员工最终使用的是福利站的统一兑换规则。

## 4. 福利站的典型功能

### 4.1 员工端

- 员工登录和身份认证；
- 查看积分余额或福利额度；
- 浏览福利商品；
- 根据分类、价格、积分和活动筛选商品；
- 使用积分或“积分 + 现金”兑换商品；
- 查看兑换记录、订单和物流；
- 管理收货地址；
- 使用优惠券、员工专属权益或活动权益；
- 提交售后和客服咨询。

### 4.2 企业管理端

- 员工和组织架构管理；
- 福利预算和积分规则配置；
- 节日、生日、奖励等福利活动配置；
- 商品和供应商管理；
- 福利商品上下架；
- 员工福利发放和审批；
- 订单、库存、物流和售后管理；
- 积分流水、预算和运营数据统计。

### 4.3 平台运营端

- 供应商接入和管理；
- 商品采集、标准化和属性维护；
- 商品库存、价格和履约管理；
- 订单分发和供应商结算；
- 客服、售后和异常处理；
- 商品运营和活动运营。

## 5. 当前 Glodex Agent 项目是什么

当前项目是一个商品搜索和智能购物 Agent 后端，同时配套了前端对话控制台。

当前项目已经具备以下能力：

- AgentLoop 和多轮任务编排；
- 商品搜索和候选商品召回；
- 京东、淘宝、拼多多等不同平台的商品来源标识；
- 向量召回和个性化召回；
- 商品属性抽取和标准化；
- 商品筛选、商品比较和购物总结；
- 用户偏好和长期记忆；
- FastAPI 服务接口；
- AGUI/WebSocket 实时事件推送；
- 前端对话式控制台。

当前的 `item_search` 主要面向普通电商购物场景，其 `platform` 字段表示外部电商平台，例如 `jd`、`taobao`、`pdd`。它还不是一个完整的福利商城交易系统。

## 6. 希望接入福利站的功能

目标不是重新开发一个福利商城，而是把 Glodex Agent 作为福利站中的“AI 福利助手”嵌入现有系统。

员工可以通过自然语言使用福利站：

```text
员工：我有 800 积分，帮我找一份适合送父母的礼物。

AI：查询员工积分、福利商品、库存和兑换规则，
    推荐符合条件的商品，并解释推荐理由。

员工：帮我兑换第一件。

AI：展示兑换商品、所需积分、收货地址和订单信息，
    员工确认后调用福利站原有兑换接口完成订单创建。
```

### 6.1 AI 福利助手的目标能力

- 用自然语言搜索福利商品；
- 按积分余额筛选可兑换商品；
- 根据员工偏好推荐商品；
- 比较多个福利商品和供应商；
- 解释商品属性、兑换规则和限制条件；
- 查询积分余额和积分有效期；
- 查询兑换订单和物流状态；
- 生成节日、生日或家庭场景的福利组合；
- 在员工确认后跳转或调用福利站兑换流程。

## 7. 接入后的系统定位

福利站继续作为业务主系统，Glodex 作为 AI 交互和智能决策辅助层。

```text
福利站主系统
├── 员工登录和企业身份
├── 组织和员工管理
├── 积分账户
├── 福利商品中心
├── 库存和供应商
├── 订单和兑换
├── 物流和售后
└── AI 福利助手
       ↓
   Glodex Agent
       ├── 意图理解
       ├── 福利商品搜索
       ├── 个性化推荐
       ├── 商品比较
       ├── 积分和规则问答
       └── 兑换流程引导
```

## 8. 如何处理当前的多平台商品搜索

当前项目的 `item_search` 支持京东、淘宝、拼多多等平台。接入福利站后，不建议把福利商品直接伪装成某一个外部平台，也不建议在员工兑换场景中直接实时搜索外部电商商品。

建议增加一个独立的搜索范围或商品目录概念：

```text
普通电商场景
  scope = external
  platform = jd / taobao / pdd / all

福利商城场景
  scope = welfare
  catalog = employee_benefits
```

调用关系可以是：

```text
普通用户购物
    → 现有多平台 item_search

企业员工福利购物
    → welfare item_search
    → 福利站统一商品池
    → 积分、库存和员工资格过滤
```

福利商品的真实来源可以继续保留：

```json
{
  "item_id": "welfare_001",
  "title": "坚果礼盒",
  "catalog": "employee_benefits",
  "source_provider": "供应商A",
  "source_platform": "taobao",
  "market_price_cny": 128,
  "points_price": 800,
  "stock": 100,
  "redeemable": true
}
```

其中：

- `catalog` 表示商品属于福利商城；
- `source_provider` 表示实际供应商；
- `source_platform` 表示后台商品来源；
- `points_price` 表示员工兑换所需积分；
- `redeemable` 表示当前员工是否可以兑换。

员工看到的是福利站统一商品，不需要关注商品来自哪个外部平台。

## 9. 建议新增的福利数据字段

当前商品模型主要使用人民币价格。福利场景需要在标准商品字段基础上增加：

```text
points_price          积分兑换价
cash_price            需要补交的现金金额
redeemable             是否可兑换
stock                  福利库存
source_provider        供应商
source_platform        原始渠道
catalog                商品目录
tenant_id              企业租户
benefit_category       福利分类
eligibility_rule       员工资格规则
exchange_limit         员工兑换数量限制
valid_from / valid_to  福利有效期
```

## 10. 建议对接的福利站接口

Glodex 不应直接操作福利站数据库，应该通过福利站提供的业务 API 接入。

### 10.1 员工和积分

```text
GET /employee/profile
GET /employee/points
GET /employee/points/records
```

### 10.2 福利商品

```text
GET /benefits/products
GET /benefits/products/{product_id}
GET /benefits/products/{product_id}/eligibility
```

### 10.3 订单和兑换

```text
POST /benefits/orders/preview
POST /benefits/orders
GET /benefits/orders
GET /benefits/orders/{order_id}
POST /benefits/orders/{order_id}/cancel
```

### 10.4 物流和售后

```text
GET /benefits/orders/{order_id}/logistics
POST /benefits/orders/{order_id}/after-sales
```

实际接口名称以福利站现有系统为准。Glodex 需要增加一个福利站 API Adapter，将现有接口转换成 Agent 可以调用的工具。

## 11. 交易安全边界

AI 可以负责：

- 查询员工信息；
- 查询积分和积分规则；
- 搜索和推荐福利商品；
- 比较商品；
- 生成兑换预览；
- 查询订单和物流；
- 引导员工完成兑换。

AI 不应该自行决定或直接修改：

- 积分扣减；
- 库存扣减；
- 员工资格；
- 订单最终确认；
- 退款和积分返还；
- 企业福利预算。

这些操作必须由福利站后端进行最终校验，并且兑换动作应当经过员工明确确认。

推荐的兑换流程：

```text
员工提出兑换请求
        ↓
AI 查询福利站并生成兑换预览
        ↓
展示商品、积分、地址和限制条件
        ↓
员工明确确认
        ↓
福利站后端再次校验积分、库存和资格
        ↓
创建订单并扣减积分
```

## 12. 推荐的接入方式

### 12.1 页面嵌入

在福利站首页、商城页或商品详情页增加“AI 福利助手”入口。

可以采用：

- 悬浮聊天窗口；
- 独立 AI 助手页面；
- 商品详情页“问问 AI”；
- 福利首页的智能推荐区域。

### 12.2 身份传递

员工从福利站进入 AI 助手时，由福利站传递短期身份令牌。Glodex 使用该令牌调用福利站接口，不要求员工再次登录。

```text
福利站登录
    ↓ 生成短期 token
AI 助手接收 token
    ↓
查询当前员工积分、资格和订单
```

不建议把员工手机号、积分余额等敏感信息直接放在前端 URL 中传递。

## 13. 分阶段实施建议

### 第一阶段：只读智能助手

目标是先验证 AI 能力，不修改现有交易流程。

- 接入员工身份；
- 查询积分余额；
- 查询福利商品；
- 按积分和规则筛选；
- 个性化商品推荐；
- 查询订单和物流；
- 点击商品后跳回福利站详情页。

### 第二阶段：兑换预览

- 查询商品兑换规则；
- 计算积分和现金差额；
- 校验库存和员工资格；
- 展示收货地址和订单预览；
- 由员工确认后进入原有兑换页面。

### 第三阶段：对话式兑换

- AI 调用订单预览接口；
- 员工确认后调用创建订单接口；
- 处理积分扣减、库存和订单结果；
- 支持订单查询、物流和售后。

## 14. 结论

福利站是企业员工福利的业务主系统，负责员工、积分、商品、供应商、订单和履约；Glodex Agent 适合作为嵌入式 AI 福利助手，负责自然语言交互、商品搜索、个性化推荐、福利规则问答和兑换引导。

当前 `item_search` 的京东、淘宝、拼多多能力可以继续保留，但需要新增 `welfare` 搜索范围和福利商品适配层。福利场景不能直接把外部电商搜索结果当成可兑换商品，而应优先搜索福利站统一商品池，并通过福利站接口校验积分、库存和员工资格。

最终项目定位为：

> **在现有企业福利站中嵌入 Glodex AI 福利助手，让员工可以用自然语言查福利、选商品、比商品、问规则，并在确认后完成积分兑换。**
