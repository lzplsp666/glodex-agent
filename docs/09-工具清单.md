# 九个工具在 AgentLoop 中的位置

Globex 项目里，主 AgentLoop 可以调用九个业务工具。这些工具不是“子智能体”，而是主 loop 循环里的行动选项。

AgentLoop 每一轮按照 Think → Act → Observe → Reflect 运转：

- Think：判断当前用户意图、缺失信息和下一步行动。
- Act：调用一个工具，或通过 dispatch_tool 派发同质子 AgentLoop。
- Observe：读取工具返回的结构化结果。
- Reflect：判断信息是否足够，足够则进入最终总结，不足则继续下一轮。

## 工具清单

| 工具名 | 归属 | 什么时候被主 loop 调用 |
| --- | --- | --- |
| Planner | 内部 | Think 阶段判断“需求复杂，需要拆解”时 |
| ChatFallback | 内部 | Think 阶段判断“这不是购物意图，是闲聊”时 |
| WebSearch | 外部 | Think 阶段判断“需要外部事实补全”时 |
| CategoryInsight | 外部 | Think 阶段判断“需要品类趋势 / 爆款数据”时 |
| ItemSearch | 外部 | Think 阶段判断“需要检索商品”时 |
| ItemPicker | 内部 | Reflect 阶段判断“搜回来的商品需要二次筛选”时 |
| PriceCompare | 外部 | Reflect 阶段判断“需要跨平台比价”时 |
| ShippingCalc | 外部 | Act 阶段计算“关税运费”时，通常和 ItemSearch 配合 |
| ShoppingSummary | 内部 | Reflect 阶段判断“信息足够，可以生成采购清单”时 |

## 内部工具与外部工具

这里有两个容易混淆的点。

**内部工具** 不需要调用外部 API，主要靠模型自身能力或已有上下文完成任务。例如：

- Planner 本质是把自然语言需求转成结构化字段。
- ChatFallback 本质是普通对话兜底。
- ItemPicker 本质是根据用户偏好和候选集做二次过滤。
- ShoppingSummary 本质是把已有结果整理成最终清单。

**外部工具** 需要调用真实搜索引擎、比价接口、电商平台 API、物流税费服务或 RAG 知识库。例如：

- WebSearch 查询公开网页事实。
- CategoryInsight 查询品类趋势、爆款数据或商品知识库。
- ItemSearch 查询平台商品。
- PriceCompare 查询或计算跨平台价格。
- ShippingCalc 估算关税和运费。

## 与 dispatch_tool 的关系

dispatch_tool 不是九个业务工具之一，而是 fork 同质子 AgentLoop 的调度工具。

主 AgentLoop 发现某个任务适合并行、隔离上下文或需要多轮调用链时，可以通过 dispatch_tool 派发子 AgentLoop。子 AgentLoop 仍然使用同一套 LLM、同一套提示词和同一批工具，只是拥有独立的 sub_thread_id。

典型场景：

- 同时在多个平台执行 ItemSearch。
- 对大量候选商品做独立筛选。
- 某个子任务自己还需要 Think → Act → Observe → Reflect 多轮循环。

## 设计原则

1. 主 AgentLoop 只负责判断下一步该调用哪个工具。
2. 工具只负责完成单一业务动作，不自己决定整体任务流程。
3. 外部工具必须通过 monitor 上报开始、结束、错误等 AGUI 事件。
4. 内部工具也可以上报事件，方便前端展示 Agent 的推理进度。
5. ShoppingSummary 是终结性工具，一旦信息足够，应优先生成最终采购清单。
