# 技术栈与依赖清单

## 1. Python 包

### 1.1 API 层

| 包 | 用途 | 必选 |
|----|------|:--:|
| `fastapi` | HTTP 路由 + WebSocket 端点 | ✅ |
| `uvicorn[standard]` | ASGI 服务器，跑 FastAPI | ✅ |

WebSocket 直接用 FastAPI 内置的 `WebSocket` 类，不额外装 `websockets` 包。

### 1.2 Agent 层

| 包 | 用途 | 必选 |
|----|------|:--:|
| `langgraph` | Agent 图编排（StateGraph + 条件路由 + Checkpointer） | ✅ |
| `langchain` | 消息类型、工具定义基类 | ✅ |
| `langchain-openai` | ChatOpenAI 封装，兼容所有 OpenAI 格式 API | ✅ |

三个包的层级关系：

```
langgraph           ← 图编排（build graph → compile → invoke/astream）
  └─ langchain      ← 消息/工具基类（HumanMessage, ToolMessage, BaseTool）
       └─ langchain-openai  ← LLM 连接（ChatOpenAI，换供应商只改 base_url）
```

**⚠️ 版本号待定**。三个包共享 `langchain-core` 底座，版本必须互洽，否则 pip 安装时会互相覆盖导致不兼容。需确认具体版本号后再锁定范围（策略：`>=X.Y.Z,<主版本+1`）。

### 1.3 工具层

| 包 | 用途 | 必选 |
|----|------|:--:|
| `httpx` | 异步 HTTP 客户端，工具调外部 API（商品搜索、比价） | ✅ |
| `pydantic` | 工具入参/出参模型校验（FastAPI 自带，不额外装） | ✅ |

Agent 跑在 asyncio 事件循环里，工具调外部 API 必须用异步 `httpx`，不能用同步 `requests`（会阻塞整个 AgentLoop）。

### 1.4 Memory / Recall 层

#### Embedding

**评测基准**：[MTEB](https://huggingface.co/spaces/mteb/leaderboard)（Massive Text Embedding Benchmark）是全球 embedding 模型权威排行榜，覆盖 8 类检索/分类任务。中文场景看 [C-MTEB](https://github.com/FlagOpen/FlagEmbedding)，专测中文 embedding。选择时优先看 **Retrieval** 项的得分——这是 Memory/Recall 向量召回最核心的指标。

**不同场景推荐**：

| 场景 | 推荐方案 | 理由 |
|------|---------|------|
| 英文通用，零运维诉求 | `OpenAI text-embedding-3-small/large` | MTEB 英文前列，API 调一下就行，按 token 计费 |
| 中文场景，效果优先 | `BGE-M3` 或 `BGE-large-zh-v1.5`（智源） | C-MTEB 中文常年前三，开源，本地/云都能跑 |
| 中文场景，已有阿里云百炼 | `阿里云 DashScope embedding API` | 跟 LLM 同一个 base_url + api_key，零额外配置 |
| 数据不出内网 | `sentence-transformers` + 加载 `BGE` 模型 | 纯本地，不传数据出去，吃 GPU |
| 多语言（中英混合） | `BGE-M3` 或 `Cohere embed-multilingual` | 两种都原生支持多语言，MTEB 多语榜并列前茅 |
| 快速跑通，不想选 | 哪个供应商 LLM 在用，就用它家的 embedding | 一张账单，一套 key，管理成本最低 |

**选择优先级**（按实际情况排）：

1. LLM 供应商自带 → 零额外成本（你的项目：阿里云百炼）
2. 评测榜对应语言前三名 → 效果兜底
3. 运维能力 → 没 GPU 就别碰本地模型
4. 预算 → 免费（本地 BGE）vs 按量付费（API）

#### 向量数据库

Phase 1 开发用 ChromaDB 嵌入式快速跑通，生产再切。

| 引擎 | 出身 | 核心特点 | 优点 | 缺点 |
|------|------|---------|------|------|
| **Milvus** | 纯向量检索 | 向量索引种类最多（IVF / HNSW / DiskANN），≥2.5 支持稀疏向量做混合检索 | 向量性能最强，比 ES 省内存，部署比 ES 轻 | 全文检索是后加的，中文分词要自己接 |
| **Elasticsearch** | 全文检索 | 倒排索引天花板，≥8.0 加向量检索 | 中文分词接个 IK 分词器就搞定，生态最老最全 | JVM 吃内存（起步 2G），部署重，向量检索性能不如 Milvus |
| **OpenSearch** | ES 7.10 开源分支（AWS 主导） | 功能跟 ES 对等，2.x 起支持向量 k-NN | 跟 ES 差不多，协议纯 Apache 2.0 | 跟 ES 差不多重，国内用得少 |
| **ChromaDB** | 嵌入式向量库 | pip 装即用，零运维 | 最轻，开发阶段最快上手 | 单机，量大了扛不住，动态索引弱 |

实际选择逻辑：

```
场景判断
  ├── 向量为主（语义召回，本项目 Memory / Recall）
  │     → 选 Milvus。向量性能最硬，混合检索够用，简历加分
  │
  ├── 全文为主（海量商品自己建索引做关键字搜索）
  │     → 选 ES。倒排索引独一档，中文分词生态好
  │
  └── 开发阶段 / 数据量小
        → 选 ChromaDB。先跑通，后面切 Milvus 几乎无感
```

**为什么 Milvus 和 ES 在国内最主流**：两者都是生产验证过的，出了问题社区有答案，简历上也好写。OpenSearch 国内基本不用，Weaviate/Qdrant 太小众。
