## 1. 设计理念

### 1.1 分层压缩策略

```
第一层：工具结果截断 (P0)
    ↓ 防止超大结果直接进入上下文
第二层：滑动窗口 (P1)
    ↓ 保留最近 N 轮对话，丢弃旧消息
第三层：摘要压缩 (P2)
    ↓ 用 LLM 将历史消息压缩成摘要
```

### 1.2 触发时机

| 时机 | 位置 | 作用 |
|------|------|------|
| **pre_reasoning** | 推理前 | 检查 token 预算，需要时压缩 |
| **post_acting** | 工具调用后 | 截断超大工具输出 |
| **pre_reply** | 回复前 | 最终检查和整理 |

### 1.3 降级策略

```
LLM 摘要压缩失败
    ↓ 降级
滑动窗口截断
    ↓ 降级
简单的消息数限制
```

## 3. 接口定义

### 3.1 ContextCompressor（核心压缩接口）

```java
/**
 * 上下文压缩器接口
 * 
 * 参考 QwenPaw 的 BaseContextManager 设计
 */
public interface ContextCompressor {

    /**
     * 压缩消息列表
     * 
     * @param messages 原始消息列表
     * @param config 压缩配置
     * @return 压缩后的消息列表
     */
    List<Message> compress(List<Message> messages, CompressionConfig config);

    /**
     * 压缩单个工具调用结果
     * 
     * @param result 工具调用结果
     * @param maxTokens 最大 token 数
     * @return 压缩后的结果
     */
    String compressToolResult(String result, int maxTokens);

    /**
     * 生成消息摘要
     * 
     * @param messages 要摘要的消息列表
     * @param previousSummary 上一次的摘要（可选，用于增量更新）
     * @return 摘要消息
     */
    Message summarize(List<Message> messages, String previousSummary);
}
```

## 4. 实现类详解

### 4.1 DefaultContextCompressor（默认压缩器）

```java
/**
 * 默认上下文压缩器
 * 
 * 组合三种压缩策略：截断 → 滑动窗口 → 摘要
 */
@Component
public class DefaultContextCompressor implements ContextCompressor {

    private final TokenCounter tokenCounter;
    private final ChatClient llmClient;

    public DefaultContextCompressor(TokenCounter tokenCounter, ChatClient llmClient) {
        this.tokenCounter = tokenCounter;
        this.llmClient = llmClient;
    }

    @Override
    public List<Message> compress(List<Message> messages, CompressionConfig config) {
        int currentTokens = tokenCounter.count(messages);
        
        // 如果未超过阈值，直接返回
        if (currentTokens <= config.getMaxContextTokens()) {
            return messages;
        }

        // 第一层：工具结果截断
        List<Message> truncated = truncateToolResults(messages, config);
        
        // 第二层：滑动窗口
        List<Message> windowed = applySlidingWindow(truncated, config);
        
        // 第三层：摘要压缩
        if (config.isEnableSummaryCompression() && 
            windowed.size() > config.getSummaryThreshold()) {
            return applySummaryCompression(windowed, config);
        }

        return windowed;
    }

    @Override
    public String compressToolResult(String result, int maxTokens) {
        int tokens = tokenCounter.count(result);
        if (tokens <= maxTokens) {
            return result;
        }
        
        // 截断到最大长度
        String truncated = tokenCounter.truncate(result, maxTokens);
        
        // 如果需要保留关键信息，提取关键点
        if (truncated.length() < result.length()) {
            return truncated + "\n\n[结果已截断，如需完整数据请缩小查询范围]";
        }
        
        return truncated;
    }

    @Override
    public Message summarize(List<Message> messages, String previousSummary) {
        StringBuilder content = new StringBuilder();
        
        if (StringUtils.isNotBlank(previousSummary)) {
            content.append("【历史摘要】：").append(previousSummary).append("\n\n");
        }
        
        content.append("【待总结对话】：\n");
        for (Message msg : messages) {
            content.append(msg.getRole()).append(": ").append(msg.getContent()).append("\n");
        }
        
        String prompt = """
            请总结以下对话内容，保留关键信息（问题、回答要点、工具调用结果）：
            
            %s
            
            要求：
            1. 语言简洁，不超过 500 字
            2. 保留所有关键数据和结论
            3. 不丢失工具调用的重要结果
            """.formatted(content);

        ChatResponse response = llmClient.call(
            new Prompt(List.of(new UserMessage(prompt)))
        );
        
        return new SystemMessage("[对话摘要]：" + response.getResult().getOutput().getContent());
    }

    /**
     * 截断工具结果
     */
    private List<Message> truncateToolResults(List<Message> messages, CompressionConfig config) {
        return messages.stream()
            .map(msg -> {
                if (isToolResult(msg) && msg.getContent().length() > config.getMaxToolResultTokens()) {
                    String truncated = compressToolResult(msg.getContent(), config.getMaxToolResultTokens());
                    return new Message(msg.getRole(), truncated, msg.getMetadata());
                }
                return msg;
            })
            .collect(Collectors.toList());
    }

    /**
     * 应用滑动窗口
     */
    private List<Message> applySlidingWindow(List<Message> messages, CompressionConfig config) {
        if (messages.size() <= config.getSlidingWindowSize()) {
            return messages;
        }
        return messages.subList(messages.size() - config.getSlidingWindowSize(), messages.size());
    }

    /**
     * 应用摘要压缩
     */
    private List<Message> applySummaryCompression(List<Message> messages, CompressionConfig config) {
        try {
            // 将前 N 条消息压缩成摘要
            int summaryCount = messages.size() - config.getSlidingWindowSize();
            List<Message> toSummarize = messages.subList(0, summaryCount);
            List<Message> toKeep = messages.subList(summaryCount, messages.size());
            
            Message summary = summarize(toSummarize, "");
            
            List<Message> result = new ArrayList<>();
            result.add(summary);
            result.addAll(toKeep);
            
            return result;
        } catch (Exception e) {
            // 摘要失败，降级到滑动窗口
            log.warn("Summary compression failed, falling back to sliding window", e);
            return applySlidingWindow(messages, config);
        }
    }

    /**
     * 判断是否为工具调用结果
     */
    private boolean isToolResult(Message msg) {
        return "tool".equals(msg.getRole()) || 
               msg.getMetadata().containsKey("tool_call_id");
    }
}
```

    /**
     * 告警 Agent 内存配置

    @Bean("alarmMemory")

        return new MessageChatMemoryAdvisor(
            new InMemoryChatMemory(),

        return CompressionConfig.builder()
            .maxContextTokens(maxContextTokens)
            .maxToolResultTokens(maxToolResultTokens)
            .slidingWindowSize(slidingWindowSize)
            .enableSummaryCompression(enableSummary)
            .summaryThreshold(summaryThreshold)
            .build();
    }

### 5.3 Agent 中使用

```java
/**
 * 问数 Agent 示例
 */
@Component
public class QueryAgent implements Agent {

    private final ChatClient chatClient;
    private final ChatMemory memory;
    private final ContextCompressor compressor;

    public QueryAgent(
            @Qualifier("queryMemory") ChatMemory memory,
            ChatClient chatClient,
            ContextCompressor compressor) {
        this.memory = memory;
        this.chatClient = chatClient;
        this.compressor = compressor;
    }

    @Override
    public ChatResponse chat(List<Message> messages) {
        // 1. 压缩上下文（自动触发）
        List<Message> compressed = compressor.compress(messages, getConfig());
        
        // 2. 构建 Prompt
        Prompt prompt = new Prompt(compressed);
        
        // 3. 调用 LLM
        ChatResponse response = chatClient.call(prompt);
        
        // 4. 将结果加入内存
        memory.add(response.getResult().getOutput());
        
        return response;
    }

    private CompressionConfig getConfig() {
        // 使用默认配置或根据场景自定义
        return CompressionConfig.builder().build();
    }
}
```

## 9. Cache Point 策略说明

### 9.1 概念概述

**Cache Point**（缓存点）是指在压缩过程中，某些关键信息需要被**优先保留**，不参与压缩或截断。这是一种保护机制，确保重要上下文信息不会因压缩而丢失。

### 9.2 QwenPaw 的实现方式

QwenPaw 的 `light_context_manager.py` 中通过 `_should_preserve_message()` 方法实现了简单的 Cache Point 策略：

| 保留规则 | QwenPaw 实现 | 说明 |
|---------|-------------|------|
| **系统提示词** | `msg.role == "system"` | 始终保留，不压缩 |
| **最近用户消息** | `msg.role == "user" and _is_recent(msg)` | 最近的用户消息保留 |
| **其他消息** | 默认返回 `False` | 可被压缩或丢弃 |

这种实现方式**比较基础**，主要依赖消息角色和时间窗口来判断是否保留，没有显式的"Cache Point"概念，更像是一种**隐式的保留策略**。

### 9.3 推荐的增强实现

在 Java 实现中，可以通过以下方式显式增强 Cache Point 策略：

```java
/**
 * Cache Point 保留策略
 */
public class CachePointStrategy {

    private static final int RECENT_MESSAGE_COUNT = 3;
    private final List<String> keyParamKeywords;

    public CachePointStrategy(List<String> keyParamKeywords) {
        this.keyParamKeywords = keyParamKeywords != null ? keyParamKeywords : 
            List.of("查询", "告警", "阈值", "参数", "条件");
    }

    /**
     * 判断消息是否为 Cache Point（不应被压缩）
     */
    public boolean isCachePoint(Message message, CompressionConfig config, int messageIndex, int totalMessages) {
        // 规则1：系统提示词始终保留
        if ("system".equals(message.getRole())) {
            return true;
        }
        
        // 规则2：最近的用户消息保留（最后3轮）
        if ("user".equals(message.getRole()) && isRecent(messageIndex, totalMessages)) {
            return true;
        }
        
        // 规则3：包含关键参数的消息保留
        if (config.isPreserveToolKeyInfo() && containsKeyParams(message)) {
            return true;
        }
        
        // 规则4：工具调用的最终结果保留
        if (isToolFinalResult(message)) {
            return true;
        }
        
        return false;
    }

    private boolean isRecent(int messageIndex, int totalMessages) {
        return totalMessages - messageIndex <= RECENT_MESSAGE_COUNT;
    }

    private boolean containsKeyParams(Message message) {
        String content = message.getContent();
        return keyParamKeywords.stream().anyMatch(content::contains);
    }

    private boolean isToolFinalResult(Message message) {
        return "tool".equals(message.getRole()) && 
               message.getMetadata().containsKey("is_final");
    }
}
```

### 9.4 Cache Point 在压缩流程中的位置

```
原始消息列表
    ↓
分离 Cache Point（保留）和普通消息
    ↓
普通消息进入压缩流程（截断 → 窗口 → 摘要）
    ↓
合并 Cache Point 和压缩后的普通消息
    ↓
最终压缩结果
```

### 9.5 配置建议

| Agent 类型 | Cache Point 配置 | 说明 |
|-----------|-----------------|------|
| **Router Agent** | 系统提示词 + 最近3轮用户消息 | 路由决策依赖最新意图 |
| **问数 Agent** | 系统提示词 + 查询参数 + 工具最终结果 | 查询逻辑依赖完整参数 |
| **告警 Agent** | 系统提示词 + 告警规则 + 最新告警 | 告警处理需要规则和当前状态 |
| **工单 Agent** | 系统提示词 + 工单参数 + 处理记录 | 工单流程需要完整上下文 |

## 10. 写入时机策略

### 10.1 写入时机分类

根据触发方式，记忆写入时机可分为四大类：

#### 10.1.1 定时触发

| 时机 | 说明 | 适用场景 |
|------|------|---------|
| **定时提醒** | 每隔固定时间触发（如每120秒） | 个人助理型，定期检查是否有值得保存的内容 |
| **周期性刷新** | 按固定周期（如每天、每周）触发 | 长期记忆整理，经验沉淀 |

#### 10.1.2 事件触发

| 时机 | 说明 | 适用场景 |
|------|------|---------|
| **会话结束** | 用户关闭对话或超时后触发 | 会话记忆 → 短期记忆 |
| **工具调用后** | 每次工具执行完成后触发 | 更新工具结果，任务进度追踪 |
| **子任务完成** | 子 Agent 完成任务后触发 | 向上传递结果摘要 |
| **任务开始** | 任务初始化时触发 | 写入任务目标、参数（冻结） |
| **任务结束** | 任务完成或失败时触发 | 写入任务总结、经验沉淀 |
| **Agent 间通信** | Agent 发送/接收消息时触发 | 共享上下文，关键信息传递 |
| **失败/纠错** | 任务失败或用户纠错时触发 | 更新技能定义、提示模板 |

#### 10.1.3 阈值触发

| 时机 | 说明 | 适用场景 |
|------|------|---------|
| **上下文达到阈值** | Token 数或消息数超过设定值 | 触发压缩，同时写入短期记忆 |
| **内存容量限制** | 存储达到上限时触发 | 清理旧记忆，保留关键内容 |

#### 10.1.4 主动触发

| 时机 | 说明 | 适用场景 |
|------|------|---------|
| **Agent 主动调用** | LLM 通过工具调用写入 | Agent 自主决定保存什么 |
| **用户主动保存** | 用户显式要求保存 | 用户标记重要内容 |

### 10.3 写入时机适用场景分析

| 写入时机 | 个人助理型 | 任务型多 Agent | 优缺点 |
|---------|-----------|---------------|-------|
| **定时提醒** | ✅ 适用 | ❌ 不适用 | 优点：定期整理；缺点：浪费资源，可能写入无关内容 |
| **会话结束** | ✅ 适用 | ✅ 适用 | 优点：完整保存；缺点：实时性差 |
| **工具调用后** | ✅ 适用 | ✅ 适用 | 优点：实时更新；缺点：写入频繁，可能产生噪声 |
| **子任务完成** | ❌ 不适用 | ✅ 适用 | 优点：进度追踪；缺点：需要任务拆解 |
| **任务结束** | ❌ 不适用 | ✅ 适用 | 优点：完整总结；缺点：中间状态丢失 |
| **上下文达到阈值** | ✅ 适用 | ✅ 适用 | 优点：自动触发；缺点：可能丢失关键信息 |
| **Agent 主动调用** | ✅ 适用 | ✅ 适用 | 优点：精准保存；缺点：依赖 LLM 判断 |

### 10.4 通用写入方案推荐

#### 10.4.1 方案设计

推荐一个**可配置的通用写入策略**，根据 Agent 类型选择不同的写入时机组合：

```java
/**
 * 通用写入时机配置
 */
@Data
@Builder
public class WriteTimingConfig {

    @Builder.Default
    private boolean onTaskStart = true;

    @Builder.Default
    private boolean onToolCall = true;

    @Builder.Default
    private boolean onSubTaskComplete = false;

    @Builder.Default
    private boolean onThreshold = true;

    @Builder.Default
    private boolean onTaskEnd = true;

    @Builder.Default
    private boolean onAgentCommunication = false;

    @Builder.Default
    private int timingIntervalSeconds = 0;

    @Builder.Default
    private boolean onFailure = false;

    @Builder.Default
    private WriteMode writeMode = WriteMode.DIRECT;

    public enum WriteMode {
        DIRECT,
        COMPRESSED,
        SUMMARY_ONLY
    }
}
```

#### 10.4.2 不同 Agent 类型的配置

| Agent 类型 | 写入时机配置 | 写入模式 |
|-----------|-------------|---------|
| **Router Agent** | `onTaskStart` + `onTaskEnd` + `onThreshold` | `DIRECT` |
| **问数 Agent** | `onToolCall` + `onTaskEnd` + `onThreshold` | `COMPRESSED` |
| **告警 Agent** | `onToolCall` + `onSubTaskComplete` + `onTaskEnd` | `DIRECT` |
| **工单 Agent** | `onTaskStart` + `onToolCall` + `onTaskEnd` + `onAgentCommunication` | `COMPRESSED` |

#### 10.4.3 写入流程

```
任务开始 → onTaskStart → 写入任务目标（冻结）
    ↓
工具调用 → onToolCall → 写入工具结果（实时更新）
    ↓
子任务完成 → onSubTaskComplete → 写入子任务摘要（向上传递）
    ↓
上下文达阈值 → onThreshold → 压缩 + 写入短期记忆
    ↓
任务结束 → onTaskEnd → 写入任务总结（沉淀）
```

#### 10.4.4 关键设计要点

| 设计点 | 说明 |
|-------|------|
| **写入模式选择** | DIRECT：直接写入；COMPRESSED：压缩后写入；SUMMARY_ONLY：只写入摘要 |
| **批量写入** | 多个工具调用结果可批量合并写入，减少 IO |
| **异步写入** | 非关键写入可异步执行，不阻塞主流程 |
| **写入前检查** | 检查是否需要写入（去重、过滤噪声） |
| **失败重试** | 写入失败时重试，确保数据不丢失 |

### 10.5 与压缩时机的配合

写入时机和压缩时机需要配合使用：

```
压缩时机 → 控制上下文大小
    ↓
写入时机 → 控制何时持久化

配合方式：
1. 压缩后写入：先压缩再写入，减少存储占用
2. 写入前检查：写入前检查是否需要压缩
3. 增量写入：只写入新增内容，避免重复
```

---

### 10.6 总结

| 写入时机 | 核心优势 | 适用场景 |
|---------|---------|---------|
| **定时触发** | 定期整理 | 个人助理型长期记忆 |
| **事件触发** | 实时响应 | 任务型多 Agent |
| **阈值触发** | 自动控制 | 所有类型 |
| **主动触发** | 精准保存 | 所有类型 |

**通用方案**：以**事件触发为主**（任务节点），**阈值触发为辅**（自动压缩），**主动触发为补充**（精准保存）。根据 Agent 类型配置具体的写入时机组合。
