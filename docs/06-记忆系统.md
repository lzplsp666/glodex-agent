# 记忆系统

## 一、记忆的类型

Agent 的记忆从时间维度分三层：

| 类型 | 存储位置 | 生命周期 | 内容 |
|------|---------|---------|------|
| **工作记忆**（Working） | AgentLoop State messages | 当前会话 | 本轮对话历史、工具调用结果 |
| **短期记忆**（Short-term） | LangGraph Checkpointer | 跨请求，会话内 | 当前会话完整状态快照 |
| **长期记忆**（Long-term） | Memory Store（向量库） | 跨会话，持久化 | 用户偏好、历史行为、关键事实 |

从内容维度分：

| 类型 | 说明 | 示例 |
|------|------|------|
| **用户偏好** | 用户的长期喜好和约束 | "不喜欢入耳式""预算通常 200-300""偏好京东" |
| **事实记忆** | 已确认的客观信息 | "上次搜蓝牙耳机，A 款最低价在京东 189" |
| **经验记忆** | 历史决策和结果 | "上次比价场景 3 次工具调用出结果，这次可以复用" |
| **会话摘要** | 过往会话的压缩版 | "上个月用户买了跑步耳机，这次要买办公耳机" |

## 二、上下文压缩策略

Agent 跑久了 State 里 messages 会爆炸（几十轮对话 + 工具结果），必须压缩。五种策略：

### 2.1 滑动窗口（Sliding Window）

保留最近 N 条消息，旧的全扔。

**优点**：实现最简单。**缺点**：失忆——扔掉的可能有关键信息（用户最早说的预算限制）。

### 2.2 摘要压缩（Summarization）

旧消息不扔，用 LLM 压缩成一段摘要。

```
前 20 轮对话压缩为：
"用户想买蓝牙耳机，预算 200，偏好金属材质、商务风，已排除索尼和 Beats"
```

**优点**：信息不丢。**缺点**：摘要会漏细节，多一层 LLM 调用有延迟和费用。

### 2.3 关键消息保留（Selective Retention）

不全部扔也不全部压，挑重要的保留。判断维度：

- 用户明确说过"记住"的内容
- 被后续工具调用引用过的消息
- 最后一条 user message 附近的上下文
- LLM 自己判断"这条后面可能还要用"

**优点**：精准度最高。**缺点**：判断逻辑复杂。

### 2.4 混合策略（Hybrid）

前一段摘要 + 最近 N 条原文。大多数 Agent 项目的默认方案。

```
[摘要: 用户偏好...] [最近 5 轮对话原文] [当前输入]
```

### 2.5 结构化记忆提取（Structured Extraction）

不把旧对话压成文字，而是提取成结构化字段存到 Memory Store。

```
"我不喜欢入耳式，戴着疼"
  → 提取: { type: "偏好", category: "耳机类型", value: "非入耳式", reason: "佩戴不适" }
```

后续 Recall 检索直接命中这条结构化记录，不用在摘要里大海捞针。

## 三、KV Cache 对压缩策略的约束

### 3.1 大模型 KV Cache 机制

大模型厂商（OpenAI、DeepSeek、阿里百炼等）的 KV Cache 按**前缀匹配**：

```
请求 1: [系统提示词] [消息1] [消息2] [消息3]  → 算 KV，缓存
请求 2: [系统提示词] [消息1] [消息2] [消息3] [消息4]  → 前 4 条命中缓存，只算消息4
请求 3: [系统提示词] [消息1] [消息2] [消息3] [消息4] [消息5]  → 全命中，只算消息5
```

前缀不变，只往后追加，Cache 全命中。又快又省 token 费用。

### 3.2 压缩与 Cache 的矛盾

消息超长时，五种压缩策略中四种都会破坏前缀：

| 压缩策略 | 对前缀的影响 | KV Cache |
|----------|-------------|:--:|
| 滑动窗口 | 旧消息扔掉，前缀移位 | ❌ 全废 |
| 摘要压缩 | 旧消息替换成摘要文字 | ❌ 全废 |
| 关键消息保留 | 删除部分旧消息 | ❌ 全废 |
| 混合策略 | 前缀被摘要替换 | ❌ 全废 |
| 结构化提取 | 提取存 Memory Store，原文不动 | ✅ 不影响 |

每次压缩一改前缀，KV Cache 从头算。压缩省了 token，但 Cache 废了，实际可能更慢更贵。

### 3.3 Cache Breakpoint

不动历史前缀。在前缀末端打一个断点，前面冻结永不变，新消息从断点后追加：

```
[系统提示词] [消息1..消息20]  ← 前缀冻结，KV 永久缓存
────────── Cache Breakpoint ──────────
[消息21] [消息22] ...  ← 新消息继续追加
```

**前缀只追加不修改。压缩只能发生在断点之后。** 这是唯一跟 KV Cache 共存的压缩方式。

## 四、压缩策略评估

没有万能策略，必须拿真实场景跑分对比。

### 4.1 评估维度

| 维度 | 指标 | 怎么量 |
|------|------|--------|
| **信息保真** | 压缩后关键信息丢失率 | 人工标注 N 个"必须记住"的事实，压完后看丢了多少 |
| **Token 节省** | 压缩比 | `(压缩前 token - 压缩后 token) / 压缩前 token` |
| **任务影响** | 下游任务准确率 | 用压缩后的上下文跑同一批任务，看结果跟不压缩的差距 |
| **延迟** | 压缩耗时 | 压缩本身多了一次 LLM 调用，耗时多少 |
| **KV Cache** | 缓存命中率 | 连续请求中 `cached_tokens` 占比 |

### 4.2 测试方法

```
测试集: 20 个典型购物对话（短/中/长 各若干个）
       每个对话标注 5 个"必须记住"的关键信息

对每种压缩策略:
  1. 跑完对话 → 触发压缩 → 继续跑后续对话
  2. 对比压缩前后 LLM 回答质量（人工 / LLM-as-judge）
  3. 记录 token 节省量、KV Cache 命中率、延迟

结论: 按你的场景选一个平衡点
```

### 4.3 常见取舍

| 场景 | 倾向 |
|------|------|
| 会话短（< 10 轮） | 不压缩，全保留 |
| 会话中等（10~30 轮） | 结构化提取 + 最近原文 |
| 会话长（> 30 轮） | Cache Breakpoint（前缀冻结）+ 断点后混合策略 |
| 延迟敏感 | 滑动窗口（最快，不走 LLM） |
| 精度优先 | 混合策略（摘要兜底 + 原文保细节） |
