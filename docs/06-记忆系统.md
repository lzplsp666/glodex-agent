# 记忆系统设计

本文档说明 Glodex Agent 的记忆系统第一版怎么落地。

当前结论：项目只保留一个记忆包 `app/memory`。原本预留的 `app/compress` 不再作为独立模块继续扩展，后续把它的职责迁入 `app/memory`，作为记忆系统里的“上下文压缩层”。

这样做的原因是：在 AgentLoop 里，“长期偏好”“会话摘要”“上下文压缩”“Cache Breakpoint”“工具结果截断”本质上都在解决同一个问题：让 Agent 在有限上下文内记住关键内容，同时丢掉或压缩不重要内容。因此概念上统一为记忆系统，代码上也集中在一个包，降低后续维护成本。

## 1. 总体分层

记忆系统分为五层：

| 层 | 名称 | 职责 | 代码位置 |
| --- | --- | --- | --- |
| L0 | 工具侧防线 | 工具结果进入上下文前先瘦身，避免一次返回污染整轮 AgentLoop | `app/memory/tool_guard.py` |
| L1 | Cache Breakpoint | 计算稳定区和可压缩区的分界线，保护 system prompt、当前任务、最近工具调用 | `app/memory/breakpoint.py` |
| L2 | 上下文压缩 | 对 breakpoint 之后的历史做截断、滑窗、摘要压缩 | `app/memory/compressor.py` |
| L3 | Session Memory | 当前任务的过程摘要、关键决策、阶段性结果 | `app/memory/session.py` |
| L4 | Long-term Memory | 用户长期偏好、历史纠错、稳定购买倾向，可增删查改 | `app/memory/store.py` |

这五层对应从轻到重的处理方式：

```text
工具结果截断
    -> Cache Breakpoint
    -> 上下文压缩
    -> 会话记忆
    -> 长期记忆
```

越靠前越偏工程规则，成本低；越靠后越偏语义沉淀，可能需要 LLM。

## 2. 包目录结构

计划中的 `app/memory` 目录如下：

```text
app/memory/
  __init__.py
  schemas.py
  store.py
  injector.py
  extractor.py
  policy.py
  breakpoint.py
  compressor.py
  tool_guard.py
  session.py
```

各文件职责：

| 文件 | 职责 |
| --- | --- |
| `schemas.py` | 定义记忆数据结构，例如 `MemoryItem`、`MemoryKind`、`MemoryWrite`、`CompressionResult` |
| `store.py` | 长期记忆 Store，提供增删查改和相关性检索 |
| `injector.py` | 把召回的长期记忆格式化成 system prompt 中的 `{long_term_preferences}` |
| `extractor.py` | 从用户输入、工具结果、最终总结中抽取可沉淀记忆 |
| `policy.py` | 写入策略、去重策略、过滤噪声、置信度合并 |
| `breakpoint.py` | 计算 Cache Breakpoint，决定哪些消息不能动、哪些消息可以压缩 |
| `compressor.py` | 执行上下文压缩，包含滑窗、工具结果精简、摘要压缩 |
| `tool_guard.py` | 工具结果进入上下文前的 L0 防线，控制 `item_search`、`price_compare` 等工具返回体积 |
| `session.py` | 当前任务级别的短期记忆，保存阶段摘要和关键决策 |

当前已有的 `app/compress/breakpoint.py` 和 `app/compress/compressor.py` 后续迁移到：

```text
app/memory/breakpoint.py
app/memory/compressor.py
```

迁移后 `app/compress` 可以删除，或者只保留一个临时兼容层，不再写新逻辑。

## 3. 长期记忆数据结构

第一版长期记忆只存结构化文本，不直接上向量库。

建议结构：

```python
class MemoryItem(BaseModel):
    memory_id: str
    user_id: str
    kind: Literal["preference", "constraint", "correction", "summary", "fact"]
    text: str
    scope: str = "shopping"
    tags: list[str] = []
    source_thread_id: str | None = None
    confidence: float = 0.8
    created_at: datetime
    updated_at: datetime
    deleted: bool = False
    metadata: dict[str, Any] = {}
```

常见 `kind`：

| kind | 含义 | 示例 |
| --- | --- | --- |
| `preference` | 用户软偏好 | 用户偏好小众、简约风格 |
| `constraint` | 用户硬约束 | 不要塑料、必须包邮、不要预售 |
| `correction` | 用户纠错 | 用户纠正“不要推荐拼多多” |
| `summary` | 会话沉淀摘要 | 用户本轮比较关注材质和售后 |
| `fact` | 稳定事实 | 用户常用平台是京东 |

## 4. 存储方案

第一版使用 JSONL：

```text
output/memory/user_memories.jsonl
```

原因：

1. 项目还在早期，JSONL 易调试。
2. 不引入数据库依赖。
3. 可以保留完整历史，便于排查记忆为什么被写入。
4. 后续迁移 SQLite、Postgres、Milvus 时，只替换 `store.py` 内部实现。

每条记录一行 JSON。删除默认软删除，即写入 `deleted=true` 或更新同一条记录，不立即物理删除。

## 5. CRUD 接口

`store.py` 对外暴露一个全局 `store`：

```python
store = JsonlMemoryStore(...)
```

核心接口：

```python
await store.create(
    user_id: str,
    text: str,
    kind: MemoryKind = "preference",
    tags: list[str] | None = None,
    source_thread_id: str | None = None,
    metadata: dict[str, Any] | None = None,
) -> MemoryItem

await store.write_many(
    user_id: str,
    texts: list[str],
    kind: MemoryKind = "preference",
    source_thread_id: str | None = None,
) -> list[MemoryItem]

await store.list(
    user_id: str,
    kind: MemoryKind | None = None,
    include_deleted: bool = False,
) -> list[MemoryItem]

await store.read_relevant(
    user_id: str,
    query: str,
    top_k: int = 5,
) -> list[MemoryItem]

await store.update(
    memory_id: str,
    text: str | None = None,
    tags: list[str] | None = None,
    confidence: float | None = None,
    metadata: dict[str, Any] | None = None,
) -> MemoryItem

await store.delete(
    memory_id: str,
    soft: bool = True,
) -> None
```

### 5.1 新增

新增记忆前先走 `policy.py`：

1. 过滤空文本、闲聊文本、一次性任务文本。
2. 合并近似记忆，例如“不要塑料”和“用户不喜欢塑料材质”只保留一条。
3. 如果是旧记忆的增强表达，则更新旧记忆的 `confidence` 和 `updated_at`。
4. 只有通过策略检查后才写入 JSONL。

### 5.2 删除

默认软删除：

```text
deleted = true
```

软删除的好处是可追溯，避免误删用户偏好。后续可以提供 compact 操作，把已删除记录从 JSONL 中物理清理。

### 5.3 查询

查询分两类：

1. `list`：管理用途，列出某个用户的全部记忆。
2. `read_relevant`：AgentLoop 运行时使用，根据当前 query 召回相关记忆。

第一版 `read_relevant` 不做 embedding，只做轻量相关性：

```text
query 关键词
    + memory tags
    + kind 权重
    + 文本交集
    + recency
    + confidence
```

最终最多召回 5 条，避免长期记忆把 system prompt 撑大。

### 5.4 修改

修改主要用于：

1. 用户纠错。
2. 去重合并。
3. 置信度提升。
4. 标签补全。

例如用户说“我不是不喜欢塑料，我只是不要食品接触类塑料”，就应该更新原有约束，而不是新增冲突记忆。

## 6. 记忆检索时机

长期记忆检索发生在主 AgentLoop 启动前。

流程：

```text
run_agent(query, thread_id, user_id)
    -> store.read_relevant(user_id, query)
    -> injector.format_memories(...)
    -> get_system_prompt(long_term_preferences=...)
    -> create_agent(...)
    -> agent.ainvoke(...)
```

也就是说，长期记忆先于 Planner 和所有工具调用进入上下文。

子 AgentLoop 默认不重新检索长期记忆，而是复用主 AgentLoop 已注入的 system prompt。这样可以避免子任务各自检索导致偏好不一致。

如果后续子 AgentLoop 需要更强个性化，可以在 `dispatch_tool` 中显式传入父任务已经召回的 memory snapshot，而不是让子 Agent 自己查 Store。

## 7. 记忆检索策略

记忆系统不只有压缩，也必须有检索。

压缩解决的是：

```text
当前上下文太大，怎么保留关键内容、压掉冗余内容
```

检索解决的是：

```text
用户过去已经沉淀过的信息，当前任务要不要拿出来用
```

两者都属于记忆系统，但方向不同：

| 能力 | 方向 | 主要问题 | 代码位置 |
| --- | --- | --- | --- |
| 压缩 | 从当前上下文向内收缩 | 当前对话太长怎么办 | `breakpoint.py`、`compressor.py`、`tool_guard.py` |
| 检索 | 从历史记忆向当前任务注入 | 过去记过什么，当前该用哪几条 | `store.py`、`injector.py`、`policy.py` |

### 7.1 检索什么

第一版只检索和购物决策直接相关的长期记忆：

| 类型 | 示例 | 用途 |
| --- | --- | --- |
| 用户硬约束 | 不要塑料、必须包邮、不要预售 | 传给 `ItemPicker` 做硬过滤 |
| 用户软偏好 | 偏好小众、喜欢简约、讨厌网红款 | 传给 `ItemPicker` 做加权排序 |
| 平台偏好 | 京东优先、不看拼多多 | 影响 `ItemSearch` / `Planner` 的平台选择 |
| 预算倾向 | 通常 300 元以内 | 帮助 Planner 补充默认预算判断 |
| 纠错记录 | 用户纠正过“不推荐某类商品” | 避免重复犯错 |

不检索一次性任务过程，例如：

1. 某次工具调用中间日志。
2. 已经过期的商品价格。
3. 某次搜索返回的完整候选列表。
4. 无稳定复用价值的闲聊内容。

这些内容要么进入 Session Memory，要么只在当前上下文里使用，不进入长期检索。

### 7.2 什么时候检索

检索时机分为四类：

| 时机 | 是否启用 | 说明 |
| --- | --- | --- |
| 任务开始前 | 启用 | `run_agent` 收到 query 后，先检索长期记忆，再构造 system prompt |
| Planner 后 | 后续启用 | Planner 拆出品类、预算、平台后，可二次检索更精准记忆 |
| 工具调用前 | 后续启用 | 例如 `item_picker` 前补充用户偏好 |
| 用户主动询问记忆 | 后续启用 | 用户问“你记得我喜欢什么吗”时直接查 Store |

第一版只实现“任务开始前检索”，因为它最稳定，接入成本最低。

后续如果发现召回不够准，再增加“Planner 后二次检索”：

```text
用户 query
  -> 粗检索长期记忆
  -> Planner 拆出 category / budget / platform
  -> 用结构化字段二次检索
  -> 合并记忆
  -> 注入 prompt 或传给工具
```

### 7.3 怎么检索

第一版不做向量库，先用轻量打分。

`store.read_relevant(user_id, query, top_k=5)` 的候选排序建议：

```text
score =
  文本关键词交集分
  + tags 命中分
  + kind 权重
  + confidence 权重
  + recency 权重
  - deleted / 过期 / 冲突惩罚
```

`kind` 权重建议：

| kind | 权重 | 原因 |
| --- | --- | --- |
| `constraint` | 高 | 硬约束必须优先 |
| `correction` | 高 | 用户纠错要防止重复犯错 |
| `preference` | 中 | 软偏好影响排序，但不应压过本轮明确需求 |
| `fact` | 中 | 稳定事实可辅助判断 |
| `summary` | 低 | 摘要可能比较宽泛，只作补充 |

### 7.4 检索结果怎么注入

检索结果不能原样塞进 prompt，要经过 `injector.py` 控制体积。

格式：

```text
长期记忆：
- [硬约束] 不要塑料
- [软偏好] 偏好小众、简约风格
- [平台偏好] 京东优先
```

注入规则：

1. 最多注入 5 条。
2. 每条尽量不超过 40 个中文字符。
3. 硬约束排在软偏好前面。
4. 本轮 query 明确冲突时，以本轮 query 为准。
5. 低置信度记忆可以注入为“可能偏好”，不能写成确定事实。

### 7.5 检索和压缩的关系

检索和压缩不是互相替代关系。

```text
检索：从长期 Store 里拿出少量相关记忆，放进当前任务
压缩：当前任务上下文变大后，把不重要历史压小
```

典型流程：

```text
长期记忆 Store
  -> read_relevant 检索 3-5 条
  -> injector 注入 system prompt
  -> AgentLoop 执行
  -> 工具结果进入 L0 防线
  -> 上下文过长时进入 breakpoint / compressor
  -> 任务结束后 extractor 抽取新记忆
  -> store 写回长期记忆
```

因此，记忆系统是一个闭环：

```text
检索 -> 注入 -> 使用 -> 压缩 -> 抽取 -> 写回 -> 下次再检索
```

## 8. 记忆写入时机

第一版不做定时触发，先做事件触发和任务结束触发。

| 时机 | 是否启用 | 说明 |
| --- | --- | --- |
| 任务开始 | 暂不启用 | 任务目标仍在当前上下文中，暂时不写长期记忆 |
| 工具调用后 | 部分启用 | 只对超长工具结果做 L0 截断，不直接写长期记忆 |
| 子任务完成 | 暂不启用 | 子任务结果先回到主 AgentLoop，避免多处写入 |
| 上下文达到阈值 | 启用 | 触发上下文压缩，同时可写入 Session Memory |
| 任务结束 | 启用 | 从最终结果中抽取长期偏好并写入 Store |
| 用户主动保存 | 后续启用 | 用户明确说“记住”时直接写入 |
| 用户纠错 | 后续启用 | 用户明确否定旧偏好时更新或删除旧记忆 |

第一版主要写入路径：

```text
ShoppingSummary / final result
    -> extractor.extract_memories(...)
    -> policy.filter_and_merge(...)
    -> store.write_many(...)
```

这里不要完全依赖 LangChain 最终消息里的 `additional_kwargs`。更稳的方式是 `extractor.py` 用 OpenAI 兼容模型做一次结构化抽取，要求返回 JSON。

## 9. 上下文压缩策略

上下文压缩属于记忆系统里的 L1 和 L2。

### 9.1 L0 工具侧防线

工具返回结果在进入上下文之前先控体积。

典型策略：

```text
item_search:
  默认最多返回 20 件商品
  每件商品只保留 item_id、title、price_cny、platform、rating、url
  去掉完整描述、评论原文、大图列表
  超过阈值自动截断

price_compare:
  只返回 top 3-5 个最低价结果
  不返回完整平台 API 响应

web_search:
  只保留标题、摘要、url、source
  不把整页正文塞回上下文
```

这是投入产出比最高的一层，因为它从源头减少 token，而不是等上下文膨胀之后再补救。

### 9.2 L1 Cache Breakpoint

Cache Breakpoint 是一个工程分界点，不是某个供应商 API 参数。

它把上下文分成两部分：

```text
Breakpoint 之前：稳定区，不压缩，尽量保持内容不变
Breakpoint 之后：可压缩区，可以截断、摘要或滑窗
```

稳定区保留：

1. system prompt。
2. 工具 schema 或固定工具说明。
3. 当前用户消息。
4. 最近 K 轮工具调用和工具结果。
5. 关键参数消息，例如预算、品类、硬约束。

可压缩区包括：

1. 更早的普通对话。
2. 更早的大段工具结果。
3. 已经过期的中间推理过程。

第一版函数：

```python
def compute_breakpoint(messages: list[Any], keep_recent_tool_calls: int = 3) -> int:
    ...
```

如果工具调用数量小于等于 `keep_recent_tool_calls`，则不压缩，返回 `len(messages)`。

### 9.3 L2 压缩方式

Breakpoint 之后的可压缩区按以下顺序处理：

1. 工具结果精简：超长 tool result 只保留关键字段。
2. 滑动窗口：保留最近 N 条普通消息。
3. 摘要压缩：当仍然超阈值时，用 LLM 总结旧消息。
4. 降级截断：LLM 摘要失败时，直接按消息数或字符数截断。

第一版函数：

```python
async def compress_messages(
    messages: list[Any],
    max_tokens: int,
    keep_recent_tool_calls: int = 3,
) -> CompressionResult:
    ...
```

返回：

```python
class CompressionResult(BaseModel):
    messages: list[Any]
    original_tokens: int
    compressed_tokens: int
    breakpoint_idx: int
    strategy: str
    summary: str | None = None
```

## 10. 压缩触发时机

压缩不是每轮都做，而是在必要时触发。

| 时机 | 触发条件 | 行为 |
| --- | --- | --- |
| pre_reasoning | 即将调用模型前，估算上下文超过阈值 | 计算 breakpoint，压缩可压缩区 |
| post_acting | 工具返回后，结果过大 | L0 工具侧截断或 tool result 精简 |
| pre_reply | 生成最终答复前，上下文仍过大 | 最后一次摘要或滑窗 |
| on_threshold | 会话消息数或 token 数超过阈值 | 写入 Session Memory，并压缩历史 |

当前项目第一版可以先接入两处：

1. 工具返回后：复用 `tool_guard.py` 控制工具结果体积。
2. 模型调用前：在 Agent middleware 或 `main_agent.py` 的调用前进行压缩。

## 11. OpenAI 兼容协议下的处理方式

当前项目通过 OpenAI 兼容协议接入模型：

```python
init_chat_model(
    model_provider="openai",
    base_url=os.environ["OPENAI_BASE_URL"],
)
```

因此，不直接实现 Anthropic 的 `cache_control`。

本文中的 Cache Breakpoint 只表示工程上的缓存/压缩分界线：

1. Breakpoint 前的内容尽量稳定，便于服务端 prefix cache 命中。
2. Breakpoint 后的内容允许变化和压缩。
3. 不向请求体写入 Anthropic 专属 `cache_control` 字段。
4. 如果未来使用的 OpenAI 兼容服务提供显式缓存参数，再在 LLM 适配层单独接入，不污染 `memory` 包。

换句话说：

```text
Anthropic cache_control = API 能力
Glodex Cache Breakpoint = 内部工程策略
```

两者概念相似，但落地方式不同。

## 12. 与 AgentLoop 的集成流程

主流程：

```text
用户输入
  -> 读取长期记忆 read_relevant
  -> 注入 system prompt
  -> AgentLoop Think
  -> 工具调用
  -> L0 工具结果防线
  -> Observe / Reflect
  -> 必要时触发上下文压缩
  -> ShoppingSummary 生成最终结果
  -> extractor 抽取新记忆
  -> policy 去重过滤
  -> store 写入长期记忆
```

代码接入点：

| 位置 | 接入内容 |
| --- | --- |
| `app/agent/main_agent.py` | 任务开始前读取长期记忆，任务结束后写入新记忆 |
| `app/agent/dispatch_tool.py` | 子 AgentLoop 复用父任务记忆快照 |
| `app/tools/item_picker.py` | 消费长期偏好，做商品筛选加权 |
| `app/tools/shopping_summary.py` | 输出 `learned_preferences`，供写入长期记忆 |
| `app/tools/tool_registry.py` | 后续可注册显式 `memory_tool`，支持用户主动保存/删除 |

## 13. 第一版落地顺序

建议按以下顺序实现：

1. `schemas.py`：先定义 `MemoryItem`、`CompressionResult`。
2. `store.py`：实现 JSONL 版 CRUD。
3. `policy.py`：实现基础去重和过滤。
4. `injector.py`：实现长期记忆 prompt 格式化。
5. 接入 `main_agent.py` 读路径。
6. `tool_guard.py`：先把工具结果体积控制住。
7. `breakpoint.py` 和 `compressor.py`：迁移并实现上下文压缩。
8. `extractor.py`：用 OpenAI 兼容模型抽取新偏好。
9. 接入 `main_agent.py` 写路径。
10. 写 `docs/06-记忆系统.md` 的实现补充和测试用例。

第一版优先保证：

1. 长期偏好能被写入。
2. 当前 query 能召回相关偏好。
3. 注入 prompt 的记忆足够短。
4. 工具结果不会撑爆上下文。
5. 上下文压缩失败时有降级策略。

## 14. 不做什么

第一版暂不做：

1. 向量库。
2. 定时 Memory Nudge。
3. 多 Agent 共享记忆总线。
4. Anthropic `cache_control`。
5. 复杂权限系统。
6. 自动硬删除历史记忆。

这些能力可以后续加，但不应该阻塞第一版记忆系统落地。
