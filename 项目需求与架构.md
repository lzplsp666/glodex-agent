# Glodex Agent — 项目需求与架构文档

> 基于代码分析生成的项目说明文档。本文档从实际代码出发，描述项目背景、Agent 运作流程、工具清单及多 Agent 模式。

---

## 一、项目背景

**Glodex** 是一个面向**国内电商购物场景**的 AI Agent 后端系统。核心目标是：

- 接收用户的自然语言购物需求（如"我想买一款白色内存条，预算 500 元以内，不要塑料材质"）
- 通过 AgentLoop 自主拆解意图、检索商品、比价筛选、生成推荐清单
- 支持多平台商品检索（京东、淘宝、拼多多等）
- 沉淀用户长期偏好，实现个性化推荐

**技术定位**：Python 异步后端，基于 LangChain Agent 框架，提供 FastAPI HTTP 接口 + WebSocket 实时事件推送（AGUI）。

## 三、Agent 运作流程

### 3.1 主 AgentLoop 生命周期

```
用户请求 → HTTP POST /api/task → TaskManager 创建后台任务
                                              │
                                              ▼
                                    run_agent(thread_id, user_id, query)
                                              │
    ┌─────────────────────────────────────────┼─────────────────────────────────────────┐
    │                                         │                                         │
    ▼                                         ▼                                         ▼
  检索长期记忆                          构建 System Prompt                        创建 LangChain Agent
  (store.read_relevant)                 (注入用户偏好 + 工具清单)                  (tools=FULL_TOOL_SET)
    │                                         │                                         │
    └─────────────────────────────────────────┴─────────────────────────────────────────┘
                                              │
                                              ▼
                                    agent.ainvoke({messages})
                                              │
                                              ▼
                                    ┌─────────────────────┐
                                    │  Think → Act Loop   │
                                    │  每轮 LLM 决定调用  │
                                    │  哪个工具或给出答案 │
                                    └─────────────────────┘
                                              │
                                              ▼
                                    工具执行结果回写 messages
                                    循环直到：
                                    - 调用 ShoppingSummary（终结）
                                    - 达到 30 轮上限
                                    - 用户取消 / 超时
                                              │
                                              ▼
                                    提取最终文本 → 沉淀长期记忆
                                    → 返回结果 + WebSocket 事件
```

### 3.2 决策循环（Think → Act → Observe → Reflect）

Agent 每轮遵循以下范式：

1. **Think**：LLM 分析当前状态，判断还缺什么信息
2. **Act**：调用工具（如 ItemSearch 检索商品、PriceCompare 比价）
3. **Observe**：读取工具返回的结构化数据
4. **Reflect**：信息足够了吗？够则调用 ShoppingSummary 终结；不够则回到 Think

### 3.3 任务状态流转

```
[started] → [thinking] → [tool_start] → [tool_end] → ... → [task_result]
    │            │            │            │                  │
    │            │            │            │                  ▼
    │            │            │            │            正常完成
    │            │            │            │
    └────────────┴────────────┴────────────┘
                 │
                 ▼
            [cancelled]  ← 用户主动取消 / WebSocket 发送 cancel_task
                 │
                 ▼
            [timeout]    ← 超过 300 秒
                 │
                 ▼
            [error]      ← 工具异常 / LLM 异常
```

## 四、工具清单（Tools）

所有工具均注册于 `app/tools/tool_registry.py`，主 Agent 和子 Agent **共用同一套工具集**（同质 fork）。

|--------|------|------|------|------|

| **chat_fallback** | `chat_fallback.py` | 闲聊兜底，非购物意图回复 | `user_query: str` | `ChatFallbackOutput`（reply） |

| **item_search** | `item_search.py` | 从商品索引检索候选商品（向量召回 + 本地索引回退） | `query, platform, top_k, user_id` | `ItemSearchOutput`（候选列表） |
| **item_picker** | `item_picker.py` | 从候选商品中按用户偏好精选 1-3 件 | `candidates, insight, preferences, top_n, max_budget` | `ItemPickerOutput`（精选商品 + 排除原因） |

## 五、多 Agent 模式（Fork 机制）

### 5.1 何时 Fork

主 Agent 在以下情况应调用 `dispatch_tool(demands="...")` 派发子 Agent：

1. **能并行**：多个独立检索可同时跑（如跨 4 个平台同时 ItemSearch）
2. **上下文要隔离**：子任务输出很大（如一次拉 100 件商品做精挑），避免污染主 loop
3. **调用链 ≥ 3**：子任务自己内部还要多轮 Think → Act

### 5.2 Fork 限制

- **最大深度**：2 层（`MAX_FORK_DEPTH = 2`）
- **子任务限制**：12 轮迭代，90 秒超时
- **同质性**：子 Agent 与主 Agent 共用同一 LLM、同一工具集、同一记忆快照

### 5.3 Fork 实现机制

```python
# fork_guard.py — 深度控制
with enter_fork() as depth:  # depth = 当前 fork 层级
    if depth >= MAX_FORK_DEPTH:
        raise ForkLimitExceeded  # 转成工具结果返回，不崩溃

# dispatch_tool.py — 子 Agent 创建
sub_agent = create_agent(
    model=get_llm(),           # 同一 LLM
    tools=FULL_TOOL_SET,       # 同一工具集
    system_prompt=get_system_prompt(long_term_preferences=get_memory_prompt()),
    middleware=[short_term_memory_middleware],
)
```

### 5.4 Fork 事件监控

每次 fork 会通过 WebSocket 推送事件：

```json
{
  "type": "monitor_event",
  "event": "fork",
  "message": "派发子 AgentLoop",
  "data": {
    "sub_thread_id": "sub-a1b2c3d4-d1",
    "demands": "在京东搜索白色内存条..."
  }
}
```

## 六、记忆系统

### 6.1 长期记忆（L4）

- **存储**：`JsonlMemoryStore` — 本地 JSONL 文件（`output/memory/user_memories.jsonl`）
- **结构**：`MemoryItem`（memory_id, user_id, kind, text, tags, confidence, created_at, ...）
- **类型**：`preference`（软偏好）、`constraint`（硬约束）、`correction`（用户纠错）、`summary`（会话沉淀）、`fact`（稳定事实）
- **写入时机**：任务结束后，从对话结果中提取新偏好
- **读取时机**：任务开始前，根据 query 召回 top_k=5 条相关记忆注入 system prompt
- **去重合并**：写入前扫描已有记忆，相似度 ≥75% 则合并更新

### 6.2 短期记忆（L3）

- **实现**：`ShortTermMemoryMiddleware`（LangChain AgentMiddleware）
- **功能**：
  - **工具结果截断**： oversized tool results 在进入 context 前截断（默认 8000 字符）
  - **上下文压缩**：每次 LLM 调用前检查 token 数，超限时压缩（保留最近 3 轮 tool call）
- **存储**：`session_memory` — 按 thread_id 存储会话快照

## 七、商品检索与召回

### 7.1 召回通道

| 通道 | 实现 | 说明 |
|------|------|------|
| **语义召回** | Milvus 向量 ANN | query 经 Tower 模型编码为 embedding，Milvus 近似最近邻搜索 |
| **标题召回** | Milvus scalar LIKE | 对 title 字段做关键词模糊匹配 |
| **个性化召回** | 用户 embedding 融合 | user_tower 编码用户 ID，与 query embedding 加权融合（0.6:0.4） |
| **本地索引回退** | JSON/JSONL 文件 | 未配置 Milvus 时，加载本地商品文件做关键词匹配 |

### 7.2 混合排序

```python
# hybrid_search：embedding + title 双通道，去重 boost
merged = ann_client.merge_results(
    primary=vector_hits,      # weight=1.0
    secondary=title_hits,     # weight=0.9
    duplicate_boost=0.6,      # 双通道都命中的商品额外加分
)
```

## 十三、关键设计决策

1. **同质 Fork**：主/子 Agent 共用同一套工具集和 LLM，保证行为一致性，避免工具权限不一致导致的意外。

2. **工具结果截断**：所有工具返回在进入 Agent context 前经过 `tool_guard` 截断（默认 8000 字符），防止长结果撑爆上下文。

3. **记忆双轨制**：
   - 长期记忆：跨会话持久化用户偏好（JSONL 文件）
   - 短期记忆：单会话内上下文压缩（中间件自动处理）

4. **向量召回回退**：Milvus 不可用时自动回退到本地 JSON/JSONL 关键词搜索，保证服务可用性。

5. **终结性工具**：只有 `shopping_summary` 能结束购物链路，防止 Agent 无限循环。

6. **环境变量驱动**：模型、API 地址、存储路径全部通过环境变量配置，便于不同环境切换。
