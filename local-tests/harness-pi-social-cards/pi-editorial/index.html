Pi-InspiredAgent Architecture

A FIELD GUIDE FOR BUILDING AGENTS

一个 Agent 项目
到底该怎样
分层？

From Model to Product

“把可复用的发动机，和具体业务的整车，分开。”

这不是把代码拆成更多文件。

核心问题是：哪些能力应该跨项目复用？哪些能力只属于一个业务？边界清楚，Agent 才能测试、替换、扩展。

PI STYLE / PYTHON01 / 07
Chapter 01The Pi Idea

Pi 的答案，
是三层分工。

Application LayerSession、Skills、业务 Prompt、具体产品
Agent CoreLoop、State、Events、Tool Runtime
AI Layer模型供应商、流式调用、Token

依赖方向必须向下。

Application 调用 Core；Core 依赖 AI 抽象和 Tool 接口；AI 层不应该反过来知道“商品推荐”或“客服流程”。

替换模型供应商时，只改 AI Adapter；换业务时，只改 Application。

上层定义“要做什么”，中层解决“怎样循环”，底层处理“怎样调用模型”。

PI AGENT HARNESS02 / 07
Chapter 02Directory

从一个大 agent/，
拆成六个边界。

AI
ai/：统一不同模型的调用协议
CORE
agent_core/：循环、状态、事件、工具运行时
APP
agent_app/：业务 Prompt、Session、编排
ACTION
tools/：只完成外部业务动作
MEMORY
memory/：读取、写入、压缩、长期存储
CONTROL
harness/：Hook 注册、检查、拒绝、Trace

判断归属的一个问题：这段代码换一个业务还能复用吗？

能复用 → Core / AI / Harness；业务专属 → Application / Tools；数据生命周期 → Memory。

THE DIRECTORY MAP03 / 07
Chapter 03Agent Core

Core 是发动机，
不应该懂业务。

01接收输入，维护 messages 与通用 State
02调用模型，解析 tool_call 或 final answer
03调度 Tool Runtime，回写 tool_result
04发出事件，决定继续循环还是收敛
它不需要知道“商品怎么搜索”，也不需要知道“用户是否喜欢某品牌”。这些属于工具、记忆和应用层。

Core 的接口比实现更重要

输入：Message + Config；输出：事件流 + 最终 State。模型、工具、记忆都通过接口接入，Core 才能被单测和替换。

REUSABLE RUNTIME04 / 07
Chapter 04Boundaries

工具、记忆、Harness
不是同一件事。

TOOLS
如何完成一个业务动作？返回业务结果。
MEMORY
如何存、取、压缩信息？提供上下文。
HARNESS
何时执行？是否允许？如何治理？

例：ItemSearch 返回 80 个商品

Tool 负责检索；Memory 负责偏好和历史；Harness 在 post_tool_call 截断结果、检查 Schema，并把失败写入 State。

工具做动作；记忆存信息；Harness 管运行过程。

KEEP THE BOUNDARY05 / 07
Chapter 05State

不要把所有信息，
塞进一个 State。

AgentStatemessages / model / tools / streaming / error
AgentControlStateoriginal_query / phase / budget / called_tools / drift
LongTermMemory Store跨会话偏好、历史事实、稳定决策

为什么要拆开？

Hook 不需要读完整历史，只需要稳定字段：original_query、phase、round_count、recent_actions_summary、assertions_failed。

漂移 Hook 修改 control state；Memory Store 负责持久化；Core 负责把 state 重新送回下一轮。

完整消息、业务控制和长期存储分开，Hook 才能读到稳定、紧凑、可判断的状态。

THREE KINDS OF STATE06 / 07
Chapter 06Request Path

一次请求，
穿过这些层。

INPUTapi/ 接收用户 Query
APPagent_app 恢复 Session，组装业务能力
COREAgent Loop 调用模型与工具
CONTROLHarness 在关键边界检查与纠正
OUTPUT写回记忆，返回最终结果

“先让边界清楚，复杂度才有地方安放。”
